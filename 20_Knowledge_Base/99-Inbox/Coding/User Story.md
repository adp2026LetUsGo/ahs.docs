En el mundo del desarrollo de software y la gestión de proyectos, una **User Story** (Historia de Usuario) es una parte fundamental de las **Metodologías Ágiles** (como **Scrum** o **Kanban**).

Aquí te explico exactamente dónde encaja y cuál es su jerarquía:

### 1. ¿De qué metodología forma parte?

La User Story es la unidad de medida estándar en el marco de trabajo **Scrum**. Se utiliza para definir los requisitos del software no desde una perspectiva técnica, sino desde la **perspectiva del usuario final**.

### 2. Jerarquía del Proyecto (Estructura)

Para entender dónde vive una User Story, imagina una pirámide de mayor a menor detalle:

1. **Product Vision:** El objetivo final del producto a largo plazo.
    
2. **Theme:** Un área temática muy grande (ej. "Seguridad" o "Pagos").
    
3. **Epic (Épica):** Una funcionalidad grande que no se puede terminar en un solo Sprint (ej. "Sistema de gestión de carritos de compra").
    
4. **User Story (Historia de Usuario):** Una pieza pequeña de la Épica que se puede completar en un ciclo de trabajo (Sprint).
    
5. **Task (Tarea):** El desglose técnico de la historia (ej. "Crear tabla en la base de datos", "Diseñar botón CSS").
    

### 3. La estructura clásica

Una User Story siempre sigue esta fórmula:

> **"Como [tipo de usuario], quiero [acción/objetivo] para [beneficio/valor]."**

- _Ejemplo:_ "Como cliente de una cafetería, quiero pagar con código QR para no tener que llevar efectivo."
    

### 4. ¿Cuándo se utiliza?

- **Product Backlog:** Es la lista de espera de todas las User Stories que el equipo debe construir.
    
- **Sprint Planning:** El equipo selecciona las User Stories más prioritarias del Backlog para trabajarlas en las próximas 2 semanas (el Sprint).
    
- **Definition of Done (DoD):** Una historia se considera terminada solo cuando cumple con los criterios de aceptación acordados.
    

### 5. Características (El método INVEST)

Para que una User Story sea parte efectiva de un proyecto, debe ser:

- **I**ndependent (Independiente)
    
- **N**egotiable (Negociable)
    
- **V**aluable (Aporte valor)
    
- **E**stimable (Que se pueda calcular el esfuerzo)
    
- **S**mall (Pequeña)
    
- **T**estable (Que se pueda probar/testear)
