## 5. Workflow Phases (Detailed Analysis)

---

## Phase 1: AOT Compatibility Audit

### Objective

Detect and resolve issues that would cause NativeAOT trimming or runtime failures.

### Why This Phase Exists

.NET 10 NativeAOT:

- Removes unused code aggressively
    
- Disallows late-bound reflection
    
- Requires static knowledge of used members
    

### Actions

- Execute:
    
    dotnet build /p:PublishAot=true /p:IsTrimmable=true
    
- Inspect warnings:
    
    - `IL2026` – unsafe reflection usage
        
    - `IL3050` – dynamic code generation
        

### Corrective Techniques Adopted

- Source Generators (JSON, DI, serialization)
    
- `[DynamicallyAccessedMembers]` attributes
    
- Removal of runtime type discovery
    

### Outcome

The codebase becomes **AOT-safe** and deterministic.

---

## Phase 2: Compilation Pipeline

### Objective

Produce a **self-contained NativeAOT binary** optimized for size and startup time.

### Command Used

dotnet publish -c Release -r win-x64 --self-contained true \  
  /p:PublishAot=true \  
  /p:OptimizationPreference=Size

### Critical Issue Encountered (and Resolved)

#### Problem

The Markdown code block (` ```bash `) was not properly closed during early iterations.

#### Impact

- The agent interpreted subsequent workflow text as shell commands
    
- Automation failed at runtime
    

#### Resolution

Strict Markdown correctness enforced:

- Every code block is explicitly closed
    
- Clear separation between executable commands and agent actions
    

This correction was **essential** for automation reliability.

---

## Phase 3: Binary Validation

### Objective

Ensure the produced NativeAOT binary is **functionally valid**, not just compilable.

### Why This Phase Is Mandatory in NativeAOT

NativeAOT failures often appear:

- Only at runtime
    
- After aggressive trimming
    
- With errors such as `TypeLoadException`
    

Compilation success **does not guarantee correctness**.

### Agent-Executed Actions

- Measure binary size
    
- Execute binary in a sandbox environment
    
- Detect runtime failures
    
- Log results to:
    
    .agent/plans/deployment_logs.md
    

### Key Principle

This phase transforms the build from:

> “Compiled successfully”  
> to  
> “Verified to run correctly”

---

## Phase 4: Mission Completion

### Objective

Persist execution results and close the operational loop.

### Actions Performed

- Update task status in:
    
    .agent/plans/tasks.md
    
- Publish performance metrics:
    
    - Startup time
        
    - Memory footprint
        
- Notify artifact location:
    
    ./dist/native/
    

### Why This Phase Matters

- Enables resumable agent sessions
    
- Provides auditability
    
- Prevents repeated work
    
- Converts execution into project knowledge

## Abstract

The transition of .NET toward **NativeAOT-first execution models** introduces a new class of challenges that traditional CI/CD pipelines, scripts, and human-driven processes are not designed to solve.

This whitepaper presents an **agent-driven, declarative deployment architecture** that leverages Google Antigravity to execute, validate, and persist NativeAOT build pipelines for .NET 10. ## 1. Problem Statement

### 1.1 The NativeAOT Shift

.NET 10 marks a decisive shift toward **NativeAOT** as a first-class deployment target. This shift introduces several non-negotiable realities:

- Aggressive code trimming
    
- Limited runtime reflection
    
- Elimination of JIT safety nets
    
- Runtime failures that cannot be detected at compile time
    

Traditional pipelines assume:

> “If it compiles, it works.”

In NativeAOT, this assumption is **false**.

---

### 1.2 Limitations of Traditional Pipelines

Existing build systems rely on:

- Shell scripts
    
- CI YAML
    
- Stateless runners
    
- Human interpretation of logs
    

These systems:

- Do not retain intent
    
- Do not reason about runtime failures
    
- Do not adapt to trimming-related issues
    
- Do not persist knowledge across executions
    

This creates a gap between **modern AOT constraints** and **legacy automation models**.

Instead of encoding deployment logic in scripts, we encode it in **declarative workflows** that an AI agent can:

- Remember

> “Execute blindly”

> “Execute with understanding and authorization”

Instead, the developer provides:

- **Rules** → what must always be true
    
- **Workflows** → what is allowed to happen

The agent operates strictly within these boundaries.

# 📘 Case Study

## Step-by-Step Development of `predictive-risk-project`

### A NativeAOT + Agent-Oriented Journey

---

## 1. Punto de partida: la idea inicial (no el código)

El proyecto **`predictive-risk-project`** no comenzó con una solución técnica clara, sino con una **pregunta abierta**:

> ¿Puede un agente de IA participar activamente en el ciclo de vida de un proyecto .NET moderno, especialmente bajo las restricciones de NativeAOT?

Desde el inicio había **tres tensiones**:

1. .NET 10 y NativeAOT eliminan muchas “comodidades” históricas
    
2. Los agentes (Gemini / Antigravity) **no ejecutan scripts ciegamente**
    
3. El proyecto debía ser **reproducible, auditable y autónomo**
    

Nada de esto estaba resuelto al comienzo.

---

## 2. Primer intento: pensar como CI/CD tradicional (error conceptual #1)

### Qué se intentó

El primer enfoque fue implícito (y típico):

- “Escribamos pasos”
    
- “El agente ejecutará lo que vea”
    
- “Las fases finales pueden ser documentación”
    

### Suposición errónea

> El agente entiende la intención aunque no esté explícita.

### Consecuencia

Apareció la **primera confusión fuerte del chat**:

- ¿Phase 3 y Phase 4 se ejecutan?
    
- ¿Son manuales?
    
- ¿Van en el workflow o en otro archivo?
    

Este error **no es técnico**, es **arquitectónico**.

---

## 3. Descubrimiento clave: el agente no ejecuta intención, ejecuta autorización

Aquí ocurre el **primer giro importante del proyecto**.

Se entiende que:

- El agente **no infiere permisos**
    
- El agente **no asume acciones**
    
- El agente solo actúa **si está explícitamente autorizado**
    

Esto llevó a una separación clara:

|Elemento|Rol|
|---|---|
|Rules|Leyes permanentes|
|Workflows|Autorizaciones|
|Plans|Memoria / estado|

📌 Este descubrimiento redefine todo el proyecto.

---

## 4. Segundo intento: creación del workflow `deploy-aot.md`

### Qué se hace

Se crea el archivo:

.agent/workflows/deploy-aot.md

con fases numeradas y lenguaje declarativo.

### Error técnico #1 — Markdown incompleto

#### Qué pasó

El bloque:

```bash  
dotnet publish ...

**no se cerró correctamente** en una versión intermedia del chat.

#### Log conceptual del error

- El agente interpreta:
    
    - Phase 3
        
    - Phase 4  
        como **comandos de terminal**
        

#### Causa raíz

- Markdown inválido
    
- Falta de separación entre:
    
    - comandos ejecutables
        
    - acciones declarativas
        

#### Corrección aplicada

- Cierre explícito de bloques
    
- Revisión estricta de sintaxis Markdown
    

📌 Este error demuestra que, en un sistema agent-driven,  
**la documentación ES código**.

---

## 5. Primer contacto real con NativeAOT (.NET 10)

### Acción

Se activa:

dotnet build /p:PublishAot=true /p:IsTrimmable=true

### Errores observados (esperables)

#### IL2026

- Uso de reflection no segura
    
- Serialización dinámica
    

#### IL3050

- Código generado en runtime
    
- APIs incompatibles con AOT
    

### Causa raíz

NativeAOT:

- No tolera descubrimiento tardío
    
- No permite código que “decida en runtime”
    

### Decisión tomada

- Migrar a Source Generators
    
- Anotar accesos con `[DynamicallyAccessedMembers]`
    
- Reescribir partes del dominio predictivo para ser estáticas
    

📌 Aquí el proyecto deja de ser “.NET tradicional”.

---

## 6. Error conceptual #2: “si compila, funciona”

Después de corregir warnings, el proyecto **compila en NativeAOT**.

### Suposición errónea

> El binario es válido porque no falló la compilación.

### Realidad NativeAOT

- El linker puede eliminar:
    
    - constructores
        
    - métodos usados indirectamente
        
- El error aparece **solo al ejecutar**
    

### Resultado

Se introduce **Phase 3: Binary Validation**.

No como “documentación”, sino como **obligación formal** del agente.

---

## 7. Introducción del log persistente (Planes)

Para que el proceso no se repita ciegamente, se crean:

.agent/plans/tasks.md  
.agent/plans/deployment_logs.md

### Qué problema resuelven

- El agente recuerda qué ya hizo
    
- Los errores quedan documentados
    
- El proyecto tiene trazabilidad
    

Ejemplo de log conceptual:

[NativeAOT Validation]  
- Binary size: 14.2 MB  
- Startup: OK  
- Exception: TypeLoadException (missing ctor)  
- Cause: trimmed unused generic

📌 Aquí el proyecto se vuelve **stateful**.

---

## 8. Consolidación de Phase 4: cierre de misión

Phase 4 no agrega funcionalidad técnica, pero agrega **gobernanza**.

### Acciones

- Marcar tareas completadas
    
- Publicar métricas
    
- Declarar artefacto final
    

### Importancia

Sin Phase 4:

- El agente repite acciones
    
- No hay “fin” claro
    
- No hay continuidad entre sesiones

## 9. Prácticas modernas adoptadas (resultado del proceso)

### .NET 10 / NativeAOT

- Runtime validation obligatoria
    
- Trimming como diseño, no como optimización
    
- Binarios pequeños y deterministas
    

### C# 14

- Código explícito
    
- Menos magia
    
- Más estructura
    

### Ingeniería asistida por agentes

- Autorización > automatización
    
- Documentación = contrato
    
- Logs = conocimiento del sistema
