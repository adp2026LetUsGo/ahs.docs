# 📄 **System Sequence Specification (SSS)**

## Xinfer – Pre-Dispatch Thermal Risk Assessment

---

# 1. 🎯 Scope

El sistema **Xinfer** permite:

> **Predecir el riesgo de excursión térmica de un embarque pharma ANTES del despacho**, utilizando datos históricos y condiciones operativas conocidas.

⚠️ Fuera de alcance:

- monitoreo en tiempo real
    
- telemetría en vuelo
    
- control físico del embarque
    

---

# 2. 👤 Actors

Primary Actor:  
- Logistics User (Analyst / Operator)  
  
Secondary (implicit):  
- Control Tower (orchestrates, does NOT decide)

---

# 3. 🧩 System Under Design

System: Xinfer Cell  
  
Type:  
- Autonomous Decision Cell  
- Owns:  
  - Data Readiness  
  - Historical selection  
  - Model lifecycle  
  - Prediction  
  - Explanation

---

# 4. ⚙️ Preconditions

Antes de iniciar la secuencia:

- Shipment identity defined:  
  - Product  
  - Route  
  - Date  
  - Packaging  
  
- Operational parameters provided:  
  - Carrier  
  - Departure time  
  - Estimated duration

---

# 5. 🔄 Main Success Scenario (System Sequence)

1. User → System:  
   SubmitShipmentContext(shipmentData)  
  
2. User → System:  
   SubmitOperationalParameters(operationalData)  
  
3. User → System:  
   RequestDataReadinessAnalysis()  
  
4. System → System:  
   EvaluateDatasetCompatibility()  
  
5. System → User:  
   ReturnDataReadinessStatus(status)  
   where status ∈ {Acceptable, Risky, NotAcceptable}  
  
6. [IF status = NotAcceptable]  
   System → User:  
   RejectPrediction(reason)  
   END FLOW  
  
7. System → System:  
   SelectHistoricalDataset()  
  
8. System → System:  
   EvaluateRetrainingNeed()  
  
9. [IF retrainingRequired = true]  
   System → System:  
   TrainModel()  
  
10. System → System:  
    PredictThermalRisk()  
  
11. System → System:  
    GenerateExplanation()  
  
12. System → System:  
    GenerateRecommendations()  
  
13. System → User:  
    ReturnRiskAssessment(  
        probability,  
        riskLevel,  
        explanation,  
        recommendations  
    )

---

# 6. 🔀 Alternative Flows

---

## 6.1 ❌ Data Not Acceptable (CRÍTICO)

Condition:  
DataReadinessStatus = NotAcceptable  
  
Behavior:  
- System SHALL block prediction  
- System SHALL provide actionable feedback  
  
User MUST:  
- Adjust operational parameters  
- Retry analysis

---

## 6.2 ⚠️ Risky Dataset

Condition:  
DataReadinessStatus = Risky  
  
Behavior:  
- System SHALL allow prediction  
- System SHALL flag reduced confidence

---

## 6.3 📉 Insufficient Historical Data

Condition:  
Historical dataset size < minimum threshold  
  
Behavior:  
- System MAY:  
   - degrade confidence  
   - fallback to baseline model

---

## 6.4 🔁 No Retraining Needed

Condition:  
Model is up-to-date  
  
Behavior:  
- System SHALL reuse existing model

---

# 7. 🧠 Internal System Responsibilities (CRITICAL)

Estos pasos **NO son visibles al usuario**, pero son parte del SSS:

- Dataset compatibility analysis  
- Detection of mixed/inconsistent data  
- Automatic filtering of historical records  
- Model version control  
- Training decision logic

---

# 8. 📏 Business Rules (CORE DEL SISTEMA)

---

## 8.1 🚧 Data Readiness as Mandatory Gate

RULE-01:  
System SHALL NOT execute prediction  
if DataReadinessStatus = NotAcceptable

---

## 8.2 🧠 System Thinks Before Predicting

RULE-02:  
System SHALL perform dataset validation  
BEFORE any ML inference

---

## 8.3 🤖 Historical Selection is NOT User-Controlled

RULE-03:  
System SHALL automatically select  
relevant historical data  
  
User SHALL NOT manually construct dataset

---

## 8.4 🔁 Controlled Retraining

RULE-04:  
Model retraining SHALL occur ONLY when justified  
based on:  
- new data  
- drift  
- insufficient coverage

---

## 8.5 🛑 System Can Refuse to Predict

RULE-05:  
System SHALL be allowed to refuse prediction  
if dataset is not valid

---

## 8.6 🧩 Identity vs Risk Modifiers

RULE-06:  
  
Shipment Identity:  
- Product  
- Route  
- Date  
- Packaging  
  
Risk Modifiers:  
- Carrier  
- Time  
- Duration  
- Environment

---

# 9. 📤 Postconditions

Después de ejecución exitosa:

- Risk probability calculated  
- Risk level assigned  
- Explanation generated (XAI)  
- Recommendations provided  
- Model version traceable

# 🔌 1.2 Componentes en detalle

---

## 🌐 1. API Layer

Responsabilidad:

- Exponer contratos del SSS  
- Validación básica (schema, tipos)  
- NO contiene lógica de negocio

Tecnologías típicas:

- FastAPI / ASP.NET / Spring Boot
    

---

## 🎼 2. Orchestrator (Application Service)

> 🔥 Este es el “director de orquesta”

Responsabilidades:  
  
- Ejecutar la secuencia del SSS  
- Manejar branching (Acceptable / Risky / NotAcceptable)  
- Invocar componentes internos en orden correcto

⚠️ Importante:

- NO toma decisiones de negocio profundas
    
- SOLO coordina
    

---

## 🚧 3. Data Readiness Engine (CORE)

> 💥 El componente más importante del sistema

Responsabilidades:  
  
- Validar compatibilidad del dataset  
- Detectar:  
   - falta de datos  
   - inconsistencias  
   - mezcla de contextos incompatibles  
- Clasificar:  
   - Acceptable  
   - Risky  
   - NotAcceptable

Inputs:

- shipment context
    
- operational parameters
    
- metadata histórica
    

Outputs:

{  
  "status": "...",  
  "issues": [],  
  "confidence": 0.0  
}

---

## 🧠 4. Historical Data Selector

Responsabilidades:  
  
- Filtrar datos relevantes automáticamente  
- Construir dataset de entrenamiento/inferencia

Ejemplo de lógica:

WHERE:  
- same product  
- similar route  
- similar packaging  
- similar environmental conditions

⚠️ Usuario NO participa aquí.

---

## 🔁 5. Model Manager (MLOps dentro de la Cell)

Responsabilidades:  
  
- Versionado de modelos  
- Decidir si reentrenar  
- Ejecutar entrenamiento  
- Persistir modelos

Subcomponentes:

- Training Policy Engine  
- Model Registry  
- Drift Detector (opcional)

---

## ⚡ 6. Inference Engine

Responsabilidades:  
  
- Ejecutar predicción  
- Calcular probabilidad de excursión térmica

Tecnologías:

- scikit-learn / XGBoost / PyTorch / ONNX
    

---

## 🔍 7. Explanation Engine (XAI)

Responsabilidades:  
  
- Explicar la predicción  
- Feature importance  
- Factores de riesgo

Ejemplo:

- SHAP
    
- LIME
    

---

## 💡 8. Recommendation Engine

Responsabilidades:  
  
- Traducir predicción → acción

Ejemplo:

IF high risk:  
   - change carrier  
   - adjust departure time  
   - reinforce packaging

---

## 🗄️ 9. Data Layer

Componentes:  
  
- Historical Data Store

Tecnologías:

- PostgreSQL / Data Lake / S3 / Delta Lake

## 🧾 Representación formal (PlantUML)

@startuml  
  
actor User  
  
participant "API Layer" as API  
participant "Orchestrator" as ORCH  
participant "Data Readiness Engine" as DRE  
participant "Historical Selector" as HDS  
participant "Model Manager" as MM  
participant "Inference Engine" as INF  
participant "Explanation Engine" as EXP  
participant "Recommendation Engine" as REC  
  
User -> API: SubmitShipmentContext  
User -> API: SubmitOperationalParameters  
  
User -> API: RequestDataReadiness  
  
API -> ORCH: forward request  
  
ORCH -> DRE: evaluate()  
  
DRE --> ORCH: status  
  
alt NotAcceptable  
    ORCH -> API: rejectPrediction  
    API -> User: error response  
else Acceptable/Risky  
    ORCH -> HDS: selectData()  
    ORCH -> MM: checkRetraining()  
  
    alt retrain needed  
        MM -> MM: trainModel()  
    end  
  
    ORCH -> INF: predict()  
    ORCH -> EXP: explain()  
    ORCH -> REC: recommend()  
  
    ORCH -> API: response  
    API -> User: risk assessment  
end  
  
@enduml

# CONTEXT: Xinfer – Autonomous Decision Cell

Estoy diseñando un sistema llamado **Xinfer**, una _Autonomous Decision Cell_ para predecir el riesgo de excursión térmica en embarques farmacéuticos **antes del despacho**.

---

# 🎯 OBJETIVO DEL SISTEMA

Predecir riesgo térmico usando:

- contexto del embarque
    
- parámetros operativos
    
- datos históricos relevantes
    

El sistema NO es un CRUD ni un predictor simple:  
👉 Es un sistema que **razona antes de inferir**

---

# 🧠 PRINCIPIOS CLAVE

- El sistema decide (no el usuario)
    
- Existe un **Data Readiness Gate obligatorio**
    
- El sistema puede **rechazar predicciones**
    
- La selección de datos históricos es **automática**
    
- El reentrenamiento del modelo es **condicional**
    
- Se generan:
    
    - predicción
        
    - explicación (XAI)
        
    - recomendaciones
        

---

# ⚙️ ARQUITECTURA (ALTO NIVEL)

Componentes principales:

- API Layer
    
- Orchestrator (Application Service)
    
- Data Readiness Engine ⚠️ (componente crítico)
    
- Historical Data Selector
    
- Model Manager (MLOps dentro de la Cell)
    
- Inference Engine
    
- Explanation Engine (XAI)
    
- Recommendation Engine
    
- Data Layer (historical + models)
    
- Audit & Logging (traceabilidad completa)
    

---

# 🔄 FLUJO PRINCIPAL

1. Usuario envía:
    
    - Shipment Context
        
    - Operational Parameters
        
2. Sistema ejecuta:
    
    - Data Readiness Evaluation
        
3. Si status = NotAcceptable:
    
    - Se bloquea la predicción
        
4. Si status = Acceptable/Risky:
    
    - Selección automática de históricos
        
    - Evaluación de retraining
        
    - Predicción
        
    - Explicación
        
    - Recomendaciones
        

---

# 📏 BUSINESS RULES

- NO prediction sin Data Readiness válido
    
- Usuario NO selecciona datos históricos
    
- Sistema puede negarse a predecir
    
- Modelos se versionan y controlan
    
- Todo es auditable

- probability (0–1)
    
- risk_level (Low/Medium/High)
    
- explanation
    
- recommendations
    
- model_version
